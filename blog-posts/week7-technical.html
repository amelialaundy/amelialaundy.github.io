SQL vs NoSQL Databases


What are SQL and NoSQL?

SQL- Structured Query Language is used to access and manage date held in relational database management systems (RDBMS) it is a programming language written to specifically do just this. Relational databases store information in tables, with tables being joined to access related data between the two, and each field has to be predetermined along with its data type.

NoSQL-Not Only Structured Query Language is used to access and manage data held in other means than tabular such as documents, key-value pairs, graph databases or wide-column stores. They are often used when there are high volumes of data to be stored and can use SQL query languages to access that data but don’t solely rely on it.

Pros & Cons of SQL & NoSQL

  • When creating a SQL database the various fields have to be defined beforehand, thus creating a schema that the database adheres to. So after some time when new columns need to be added a new schema has to be devised and then have all the data migrated to the new schema. The database HAS to be taken offline when the information about a new data item needs to be added as the whole database must be altered
  • NoSQL on the other hand can add information about new data items as you go without the need to take the database offline thus saving downtime and money. These are dynamic databases and can store dissimilar data together unlike SQL databases. Although in some instances it can be more difficult to add new fields to NoSQL databases, such as wide-column stores:
    “Wide Column databases stores data tables as sections of columns of data rather than as rows of data.” Source: bi-bigdata.com
  • To scale a database is to increase the size or amount of work asked of a database; such as increasing the load required of it when resources are added. SQL databases are vertically scalable, meaning they are scaled by increasing the power of the hardware as the the demand increases. Other options such as spreading SQL databases over more than one server but this is time and resource consuming. NoSQL in contrast scales horizontally which means that to increase the power of the database you can add more servers to host the database, NoSQL automatically spreads data across the various servers and so doesn’t take any engineering time in adding more resources.
  • When using transactional based applications SQL databases are more stable and promise atomicity;
    “Database transaction is collection of SQL queries which forms a logical one task. For transaction to be completed successfully all SQL queries has to run successfully. Database transaction executes either all or none, so for example if your database transaction contains 4 SQL queries and one of them fails then change made by other 3 queries will be rolled back.”
    Source: javarevisited
    They also keep the integrity of the data, atomicity is part of the transaction process where is any of the queries fail then they all fail and so do ‘rollback’ to before the queries were run. NoSQL can be used for transactional applications as they are less stable especially for more complex transactional applications due to the un-uniformed nature of them.
  • Both database types work off different but similar properties to guarantee that transactions are correctly and reliably processed; SQL works off ACID:
    • Atomicity: all or nothing, all transaction parts must succeed in order for the whole transaction to succeed.
    • Consistency: from one valid state to another, all data written to a database must be valid and ring true to the data types set in the schema of that database.
    • Isolation: concurrent execution, when multiple transactions are targeting the same data, isolation ensures that the resulting database state is the same as if the transactions had been executes one after the other (serially) otherwise they will affect the data before the other transaction has accessed the original state of that data.
    • Durability: staying committed, even if there is power loss or computer crash/errors the result of a transaction will remain so, they need to be stored permanently so that any changes to the database are met and saved rather than just assuming they are saved.
    NoSQL works off CAP:
    • Consistency: all requests see the same data, similar to Isolation where you want all transactions to see the same data before they are edited
    • Availability: each request must receive notification about the success of that request, the database is always available and responsive
    • Partition tolerance: the system or database still runs and operates as expected even if there has been failure in the system or power loss, if part of the system is unavailable, other parts are not affected
  • SQL has been around for a long time, originally created at IBM to manage data stored in their database management system. This means that there have been plenty of people before who have had an overcome the same problems that many of us may face, there is a lot of support and help readily available for using SQL. NoSQL is new, less standardised and is vendor-specific-there is no hard and fast rule about accessing, managing and maintaining NoSQL and so you are very much at the mercy of the supplier of your specific database.

In conclusion:

After dipping my toe into the sea of SQL this week I am intrigued after researching NoSQL to dip my toe in that ocean too to figure out if one of them is warmer than the other...

As a beginner I can see how the structured ways of SQL may be easier to understand and get a grip on first although I am also starting to see the cons of predefining data types as this doesn’t easily allow for future problems to arise where new data items may be needed.

NoSQL sounds fascinating in that you can store a lot of data, even petabytes of data in a database spread across many servers, allowing huge amounts of unstructured data to be stored and accessed.

I also realise that this debate is very subjective, in particular this website where the author writes with such convicted opinions you read them as true, but read down through the comments and there seems to be quite a number of people disagreeing with the points he has made…This makes it kind of confusing for a newb like me to get a handle on the pros and cons but I do realise that creating my own opinion will be based on my experience of both SQL and NoSQL so I look forward to swimming in both seas!
