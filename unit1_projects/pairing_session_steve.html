The Benefits of Pairing

Pairing is Caring - Checking in from Karim Bishay on Vimeo.

The main concepts of pairing:

  • Pairing allows two people to work together on code with one instructing and the other typing
  • The roles do change throughout the session to give each person time directing the code and typing the code
  • Pairing can be done in person on one workstation or remotely, as we have for this page, via google hangouts and using Stypi website to collaborate on the same document of code

There may be many benefits to peer pairing, some for the code itself and others for the people involved.

In terms of the code benefitting, Wikipedia claims that 'Pairs spend about 15% more time on programs than individuals. However, the resulting code has about 15% fewer defects.' thus two people can spot more errors than one working on their own. The 15% more time spent may cost a company employing them more in wages but the greatly decreased number of bugs in the code, that would normally require more expenditure to fix, results in less expenditure overall.

Another benefit would be that working together reinforces peoples relationships and makes for having a stronger communicative team. Pairing requires constant communication about every thought each has with the code, patience as well may play a part when listening to someone's idea that you might not agree with.

Pairing naturally leads to each person involved learning more skills from design to programming rules, this is what every programmer is aiming to do: learn more to grow more and create better code.

Driver

The driver writes the code into the document whilst listening to the navigator who will direct them to the exact code they want to go in the document.

Navigator

The navigator considers the code and instructs the driver on what to write, much like rally driving where the driver relies solely on the directions from the navigator in the passenger seat and won't make a turn unless it has been verbalised by the navigator.

During the remote wireframing we communicated well, and considered different factors of the design. The same goes for the coding session, we worked well with Steve as the navigator-writing the code, and myself driving.

I don't think anything didn't work, of course we had a few issues with how do we write this? or where should this go? but in terms of the actual pairing experience I think it went smoothly.
